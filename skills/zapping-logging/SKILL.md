# Zapping Logging

## Overview

Standardizes structured JSON logging in Go (zerolog) and Rust (tracing) services.
Output is always JSON to stdout ‚Üí systemd journal ‚Üí Alloy ‚Üí Loki.

**Primary goal: user journey tracking.** Every log where user context is available in scope MUST include it. This allows tracing the full lifecycle of a user across microservices in Loki.

## ‚öÝÔ∏è Logger Replacement Rules

There are two distinct cases. Apply the correct one:

### Case 1 ‚Äî Fluentd integration present ‚Üí parallel logging

If the service uses `github.com/eltelon/zapping_go_logger` (or any wrapper that sends to Fluentd), **keep it and add zerolog alongside**. Fluentd ‚Üí Elasticsearch must not be broken.

```
existing code  ‚Üí  zapping_go_logger / Fluentd  ‚Üí  Elasticsearch   (keep as-is)
new log calls  ‚Üí  zerolog ‚Üí stdout ‚Üí systemd ‚Üí Alloy ‚Üí Loki        (add alongside)
```

- **Never delete** `SendInfo()`, `SendError()`, `SendWarn()` calls or the `zapping_go_logger` import.
- **Never remove** the Fluentd dependency from `go.mod` / `go.sum`.
- Add an equivalent `log.Info().Str(...).Msg(...)` next to each existing Fluentd call.

### Case 2 ‚Äî Other Go logger present (`slog`, `logrus`, `zap`, `log`, etc.) ‚Üí replace with zerolog

If the service uses any other Go logger, **replace it entirely with zerolog**. These are just formatting libraries with no external pipeline dependency.

| Logger found | Action |
|---|---|
| `log/slog`, `golang.org/x/exp/slog` | Replace with zerolog |
| `github.com/sirupsen/logrus` | Replace with zerolog |
| `go.uber.org/zap` | Replace with zerolog |
| `log` (stdlib) | Replace with zerolog |
| Any other non-Fluentd logger | Replace with zerolog |

Steps for replacement:
1. Add `github.com/rs/zerolog` to `go.mod` if not present.
2. Create `logger.go` with the standard zerolog init.
3. Replace all calls from the old logger to zerolog equivalents.
4. Remove the old logger import and dependency from `go.mod`.

### Case 3 ‚Äî Unknown external sink ‚Üí ask

If the existing destination is something other than Fluentd (Datadog, custom sink, etc.), **stop and ask the user** before proceeding.

## ‚öÝÔ∏è Never Modify Auto-Generated Files

**SKIP these files entirely ‚Äî do not read, scan, or modify them:**

- `*.pb.go` ‚Äî protobuf message definitions generated by `protoc-gen-go`
- `*_grpc.pb.go` ‚Äî gRPC service stubs generated by `protoc-gen-go-grpc`
- Any file containing `// Code generated` in the first 5 lines
- Any file inside a directory named `protos/`, `gen/`, or `generated/`

These files are overwritten on every `protoc` run. Any changes will be lost.
If a `*.pb.go` or `*_grpc.pb.go` file contains log calls, **leave it as-is** and note it in your report.

## Detection Flow

**Tool usage:** `find` for file discovery, `grep` for reading content across files. Use these as the primary commands ‚Äî they are faster than shell loops or reading files one by one.

When invoked on a service path, run these steps in order:

**1. Detect language**

```bash
find <service_path> -name "*.go" -not -path "*/vendor/*"   # Go
find <service_path> -name "*.rs"                           # Rust
```
- `.go` files present ‚Üí Go mode (zerolog)
- `.rs` files present ‚Üí Rust mode (tracing)

**2. Detect mode**

```bash
find <service_path> -name "logger.go" -maxdepth 2   # Go
find <service_path> -name "logger.rs" -maxdepth 2   # Rust
```
- Not found ‚Üí **SETUP mode**
- Found ‚Üí **REVIEW mode**

**3. Check for existing log integrations**

```bash
# Check for Fluentd pipeline
grep -r "zapping_go_logger\|SendInfo\|SendError\|SendWarn\|fluentd" <service_path> --include="*.go" -l
grep "zapping_go_logger\|fluentd" <service_path>/go.mod

# Check for other Go loggers
grep "slog\|logrus\|go.uber.org/zap\|\"log\"" <service_path>/go.mod
```

- **Fluentd found** ‚Üí parallel logging mode: keep all existing calls, add zerolog alongside (Case 1).
- **Other Go logger found** (`slog`, `logrus`, `zap`, stdlib `log`) ‚Üí replace entirely with zerolog (Case 2).
- **Unknown external sink** (Datadog, custom, etc.) ‚Üí stop and ask the user (Case 3).

**4. Scan for user context fields in scope**

Use `grep` to locate user identity fields across the codebase before reading individual files:
```bash
grep -rn "user_id\|userId\|UserID\|uuid\|UUID\|email\|Email\|remote_addr\|RemoteAddr\b" \
  <service_path> --include="*.go" -l
```

Then for each file with matches, read it to determine which fields are in scope per function. Fields to track:

| Pattern | Maps to field |
|---------|--------------|
| `user_id`, `userId`, `UserID` | `user_id` |
| `uuid`, `UUID` | `uuid` |
| `email`, `Email` | `email` |
| `ip`, `IP`, `remote_addr`, `RemoteAddr` | `ip` |

**5. Scan all log calls**

```bash
grep -rn "log\.\|fmt\.Printf\|fmt\.Println\|SendInfo\|SendError\|SendWarn" \
  <service_path> --include="*.go"
```

For each function/handler in the codebase, determine which user identity fields are accessible in scope (parameters, local variables, struct fields, closure captures):

| Pattern | Maps to field |
|---------|--------------|
| `user_id`, `userId`, `UserID` | `user_id` |
| `uuid`, `UUID` | `uuid` |
| `email`, `Email` | `email` |
| `ip`, `IP`, `remote_addr`, `RemoteAddr` | `ip` |

**6. Categorize every log call before making changes**

Classify each log call into one of three categories:

| Category | Description | Action |
|----------|-------------|--------|
| **Missing context** | User fields ARE in scope at that line but not included in the log | **Add them** |
| **No context available** | Log occurs before any user is known (startup, DB init, config load, background workers with no user scope) | Leave as-is ‚Äî this is correct |
| **Already complete** | Log already includes all available user fields | Leave as-is |

Report the analysis before touching any file:
```
Logs missing user context (WILL FIX):
  handlers/auth.go:74  ‚Äî "login: grpc error"  ‚Äî has ip in scope, missing from log
  handlers/auth.go:86  ‚Äî "alfred returned status false"  ‚Äî has ip in scope, missing from log
  handlers/auth.go:160 ‚Äî "error saving to Redis"  ‚Äî has user_id, uuid in scope, missing from log

Logs with no user context available (correct, skip):
  main.go:12       ‚Äî "starting server"  ‚Äî startup, no user
  db/connect.go:8  ‚Äî "database connected"  ‚Äî infra, no user

Ask for confirmation before proceeding if the list is large.
```

**7. Generate or improve code** (see sections below)

**8. Re-scan after changes ‚Äî mandatory**

After applying all changes, re-read every modified file from scratch and repeat step 3-4. Look for any log call that still has user context in scope but is missing it. Do NOT rely on memory of what you already fixed ‚Äî read the actual current file content.

If new gaps are found, fix them immediately without asking. Repeat until the re-scan finds zero missing contexts.

**9. Validate** (see Validation section)

## SETUP Mode ‚Äî Go

Create `logger.go` in the package root. Do NOT include a `service` field ‚Äî Alloy sets it automatically from the systemd unit name.

```go
// logger.go
package main

import (
    "os"
    "github.com/rs/zerolog"
)

var log = zerolog.New(os.Stdout).With().
    Timestamp().
    Logger()
```

In `main.go`, ensure zerolog is imported and no conflicting log setup exists.

User context fields are added directly on each log call ‚Äî only the fields available in scope at that point:
```go
log.Info().Str("user_id", userID).Str("ip", ip).Msg("user logged in")
log.Error().Str("uuid", uuid).Err(err).Msg("failed to process request")
```

If no user fields were found, add this comment at each HTTP/NATS handler entry point:
```go
// TODO: inject user context (user_id, uuid, email, or ip) as log fields
```

## SETUP Mode ‚Äî Rust

Add to `Cargo.toml`:
```toml
tracing = "0.1"
tracing-subscriber = { version = "0.3", features = ["json"] }
```

Create `src/logger.rs`:
```rust
// logger.rs
use tracing_subscriber::fmt;

pub fn init_logger() {
    fmt()
        .json()
        .with_current_span(false)
        .with_target(false)
        .init();
}
```

In `src/main.rs`, add `mod logger;` and call `logger::init_logger()` as the first line of `main()`.

Replace `println!` log calls with tracing macros using detected user fields:
```rust
// BEFORE
println!("user {} logged in from {}", session.user_id, session.ip);

// AFTER
tracing::info!(user_id = %session.user_id, ip = %session.ip, "user logged in");
```

If no user fields found, add at handler entry points:
```rust
// TODO: inject user context (user_id, uuid, email, or ip) as tracing fields
```

## REVIEW Mode ‚Äî Go

Read all `.go` files. For each log call that is non-compliant:

**‚õî Never delete log calls.** Every existing log must be migrated to zerolog ‚Äî removing it loses observability. If a log call cannot be cleanly migrated, convert it as-is and leave a comment explaining why.

```go
// WRONG ‚Äî log deleted
- log.Println("EPG Override Middleware: Not a thumbnail or chunklist request, skipping")

// CORRECT ‚Äî log migrated to zerolog
+ log.Debug().Str("path", path).Msg("EPG Override Middleware: Not a thumbnail or chunklist request, skipping")
```

| Problem | Fix |
|---------|-----|
| `fmt.Printf` / `fmt.Println` / `log.Println` / `log.Printf` used for logs | **Migrate** to zerolog ‚Äî never delete |
| `service` field hardcoded in logger init | Remove it ‚Äî Alloy sets `service_name` automatically from the systemd unit |
| Missing `Timestamp()` in logger init | Add `.Timestamp()` to logger init chain |
| Log to `os.Stderr` | Change to `os.Stdout` |
| User fields available but not logged | Add them directly: `log.Info().Str("user_id", uid).Str("ip", ip).Msg(...)` |
| Logger chain stored in intermediate variable (`event := log.Debug()`) | Inline the full chain ‚Äî see rule below |
| Missing level | Add explicit `.Info()` / `.Error()` / `.Warn()` / `.Debug()` |
| Non-canonical key name for user field | Rename to canonical key (see table below) |

### No Intermediate Logger Variables

**‚õî Never store a logger chain in a variable.** Write the full chain inline ‚Äî the extra characters are worth the clarity.

```go
// WRONG ‚Äî hides the logger behind a variable
event := log.Debug()
event = event.Interface("sql", sql)
event.Msg("SQL query")

// CORRECT ‚Äî full chain inline
log.Debug().Interface("sql", sql).Msg("SQL query")
```

For conditional fields, use a helper function or accept that the field is always logged (with zero value) rather than fragmenting the chain:

```go
// WRONG
event := log.Info()
if userID != "" {
    event = event.Str("user_id", userID)
}
event.Msg("request")

// CORRECT ‚Äî log unconditionally when the field is in scope
log.Info().Str("user_id", userID).Msg("request")
```

### Canonical Key Names

**Scope: log calls only.** Only rename keys that appear as arguments inside a log call (`log.X()`, `zerolog`, `tracing::` macros). Never rename keys in:
- NATS payloads or message structs
- JSON serialization (`json:"..."` tags)
- Database queries or ORM fields
- HTTP request/response bodies
- Any non-logging code

If a key appears in both a log call AND a payload, rename it only in the log call. Leave the payload untouched.

**‚õî Only rename keys that appear in the left column of the table below. Any log key NOT listed there must be left exactly as-is ‚Äî do not rename, remove, or reorder it.** When in doubt, leave it alone.

**‚õî Never modify business logic.** This skill only changes: logger initialization, log call keys/fields, and import statements. It does not touch variable names, function signatures, struct definitions, control flow, or any other code.

Rename only these non-canonical variants:

| Non-canonical keys (rename these) | Canonical key |
|------------------------------------|---------------|
| `"user"`, `"userid"`, `"userId"`, `"UserID"`, `"user-id"`, `"uid"` | `"user_id"` |
| `"UUID"`, `"Uuid"` | `"uuid"` |
| `"Email"`, `"e-mail"` | `"email"` |
| `"IP"`, `"Ip"`, `"remote_addr"`, `"RemoteAddr"`, `"remote-addr"`, `"client_ip"`, `"clientIp"` | `"ip"` |
| `"lvl"`, `"log_level"`, `"severity"` | `"level"` |
| `"msg"` (zerolog uses this natively ‚Äî leave it), `"message"`, `"text"`, `"log"` | `"msg"` (Go) / `"message"` (Rust) |
| `"ts"`, `"timestamp"`, `"datetime"`, `"date"` | `"time"` |

Example:
```go
// BEFORE ‚Äî non-canonical keys
log.Debug().Int64("user", claims.UserID).Str("request_id", reqID).Msg("token validated")

// AFTER ‚Äî only "user" ‚Üí "user_id" is renamed; "request_id" is not in the table, left as-is
log.Debug().Int64("user_id", claims.UserID).Str("request_id", reqID).Msg("token validated")
```

This applies to all zerolog field methods: `.Str()`, `.Int64()`, `.Int()`, `.Uint64()`, etc.

Example transformation:
```go
// BEFORE
fmt.Printf("user %s logged in from %s\n", userID, ip)
log.Info().Msg("request received") // userID and ip available but not logged

// AFTER
log.Info().Str("user_id", userID).Str("ip", ip).Msg("user logged in")
log.Info().Str("user_id", userID).Str("ip", ip).Msg("request received")
```

## REVIEW Mode ‚Äî Rust

Read all `.rs` files. For each non-compliant log:

| Problem | Fix |
|---------|-----|
| `println!` / `eprintln!` used for logs | Replace with `tracing::info!` / `tracing::error!` etc. |
| `tracing_subscriber::fmt().init()` without `.json()` | Add `.json().with_current_span(false).with_target(false)` |
| User fields available but not in tracing call | Add as named fields: `tracing::info!(user_id = %uid, ...)` |
| Logger not initialized in main | Add `logger::init_logger()` as first line of `main()` |
| Non-canonical field name in tracing macro | Rename to canonical key (same table as Go above) |

Example:
```rust
// BEFORE ‚Äî non-canonical keys
tracing::info!(user = %claims.user_id, request_id = %rid, "token validated");

// AFTER ‚Äî only "user" ‚Üí "user_id" is renamed; "request_id" is not in the table, left as-is
tracing::info!(user_id = %claims.user_id, request_id = %rid, "token validated");
```

Same canonical key table as Go above applies to all tracing field names.

## Loki Contract ‚Äî Required Fields Every Log Line

| Field | Source | Notes |
|-------|--------|-------|
| `time` | zerolog `Timestamp()` / tracing-subscriber | RFC3339 UTC, automatic |
| `level` | zerolog level methods / tracing level | Lowercase always |
| `message` (`msg` in zerolog) | `.Msg("...")` / `tracing::info!("...")` | Human-readable |

**Labels set by Alloy (do NOT add to code):** `role`, `instance`, `hostname`, `country`, `unit`, `service_name`, `service`

## Validation Checklist

Before declaring done, re-read all modified files and verify:
- [ ] **Re-scan complete** ‚Äî no log call with user context in scope is missing that context
- [ ] No `fmt.Printf` / `fmt.Println` / `println!` / `eprintln!` remain for logging purposes
- [ ] Every log call has an explicit level
- [ ] Timestamps configured (zerolog: `.Timestamp()`; Rust: automatic in tracing-subscriber JSON)
- [ ] Output goes to stdout (not stderr)
- [ ] All canonical key names used in log calls

**Do not ask the user "are there any remaining logs without context?" ‚Äî verify it yourself by re-reading the files.**
